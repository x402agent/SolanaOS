# SolanaOS Android App

Status: the current Android dapp is a native Solana Seeker app that keeps the SolanaOS host-shell foundation, but is now wired to the SolanaOS Go runtime, direct SolanaTracker market data, native Mobile Wallet Adapter, optional SolanaOS gateway pairing, Together-backed hosted chat, and Convex-backed user sync plus agent registry surfaces.

## What It Is Now

This is not an Expo app and it is not the older OpenClaw mobile runtime.

The shipped Android app is:

- native Kotlin + Jetpack Compose
- package id `com.nanosolana.solanaos`
- internal Android namespace `ai.openclaw.app`
- current Android version `2026.3.12`
- current release artifact base `solanaos-seeker-dappstore`

At runtime, the app can use several backplanes depending on what is configured:

- local SolanaOS Go binary on-device
- paired SolanaOS gateway over the Seeker/host connection flow
- direct SolanaTracker RPC, WS, Data API, and datastream clients
- direct xAI, Together, and OpenRouter clients for Android-side AI features
- Convex for shared identity, pairing claim state, gallery, chess archive, backend sync, and agent registry reads
- native Solana Mobile Wallet Adapter for wallet auth, message signing, and transaction signing/sending

## Architecture

### Current path

`Kotlin/Compose shell -> SolanaOS Go binary + native Android clients -> SolanaTracker / xAI / Together / OpenRouter / Convex / MWA / optional gateway`

### What stayed from the host-shell base

- Jetpack Compose shell and onboarding flow
- pairing / gateway connection model
- chat UI and streaming shell
- Android bridge surfaces for device capabilities
- foreground-service runtime model
- Seeker-oriented settings and permissions flow

### What changed

- the old Node.js/OpenClaw runtime is replaced by the SolanaOS Go binary
- the app is now Solana-first rather than a generic tool host
- market surfaces are built around SolanaTracker
- wallet flows are built around native MWA
- AI workspaces can run direct xAI / Together / OpenRouter paths on Android
- Convex is now a real shared backend surface, not just a future placeholder

## Current App Surfaces

The post-onboarding shell currently exposes these tabs:

- `Connect`
- `Solana`
- `DEX`
- `Grok`
- `Chat`
- `Arcade`
- `Voice`
- `ORE`
- `Screen`
- `Settings`

### Connect

- gateway pairing and reconnect flow
- manual host + port entry
- defaults to port `18790`
- uses the shared setup-code bundle generated by the desktop/runtime side
- exposes native Mobile Wallet Adapter wallet auth flows
- includes `Account Sync` for Convex-backed wallet identity
- includes an `Agent Registry` card that reads the signed-in wallet's NanoHub/Convex agent records
- exposes public profile, A2A, explorer, and ACP-command metadata for registered agents

### Solana

- wallet-oriented Solana runtime surface
- pairs with the local SolanaOS runtime and Android wallet state
- used as the operator dashboard for chain-aware actions

### DEX

The DEX tab is now a real SolanaTracker surface, not a placeholder token page.

It currently supports:

- latest / graduating / graduated / trending boards
- mint / symbol / name search
- focused token detail
- trades, holders, and chart data
- live slot and feed status
- direct SolanaTracker datastream and RPC websocket state
- AI token analysis through Grok and/or Convex-backed analysis paths
- swap quoting and swap execution UX tied to the connected mobile wallet

The UI explicitly treats SolanaTracker as the preferred direct market source, with Convex kept as a fallback/proxy path where needed.

### Grok

The Grok workspace is now a multi-mode Android-native AI surface:

- live web + X search
- direct prompt / response workspace
- image generation
- live camera vision

It can use:

- direct xAI when configured
- shared runtime/gateway context when connected
- SolanaTracker context for market-aware prompts
- Convex-backed gallery and shared user identity

### Chat

Chat still works with the normal SolanaOS gateway/runtime path when available.

For debug builds, the app can also fall back to direct Android-side chat when the gateway is unavailable:

- Together direct chat with `TOGETHER_API_KEY`
- default Together model `MiniMaxAI/MiniMax-M2.5`
- OpenRouter compatibility fallback when configured
- on-device key/model override via settings/build config
- clear transport state in the UI

Image attachments still depend on the live runtime/gateway path.

### Voice

Voice is now an xAI-oriented realtime voice surface rather than a generic stub.

### ORE

The old Bitaxe mobile surface has been replaced by an official ORE handoff and planner flow inside the Android app.

Current behavior:

- validates the connected Seeker wallet
- checks mainnet balance
- models an ORE round locally
- embeds the official ORE miner in-app via WebView
- keeps external browser handoff only as fallback
- shows linked Convex user state when present

This is intentionally a truthful mainnet launcher, not a fake local miner.

### Arcade

Current arcade surfaces include:

- chess
- platformer
- snake

Chess is tied into signed wallet identity and Convex-backed archive/state flows.

### Screen

- Android screen/capture-oriented runtime surface
- used as part of the broader operator shell

### Settings

- runtime, gateway, wallet, AI, and boot preferences
- includes `Start on Boot`
- foreground service restore only runs after onboarding is complete

## Build

### Full Android build flow

```bash
# Build Android ARM64 SolanaOS binary
./scripts/build-seeker.sh

# Build the APK end to end
./scripts/build-seeker-apk.sh

# Build, install, and launch from apps/android
cd apps/android
make seeker-install

# Or build directly with Gradle
cd apps/android
GRADLE_USER_HOME="$PWD/.gradle-user" ./gradlew :app:assembleDebug
```

### Install and launch on device

```bash
cd apps/android
make seeker-install
```

The helper script:

- builds `:app:assembleDebug`
- installs the newest `solanaos-mobile-*-debug.apk`
- launches `com.nanosolana.solanaos/ai.openclaw.app.MainActivity`

### Direct ADB path

```bash
cd apps/android
GRADLE_USER_HOME="$PWD/.gradle-user" ./gradlew :app:assembleDebug
adb devices
adb install -r app/build/outputs/apk/debug/solanaos-mobile-2026.3.12-debug.apk
adb shell am start -n com.nanosolana.solanaos/ai.openclaw.app.MainActivity
```

Debug APK output pattern:

```bash
apps/android/app/build/outputs/apk/debug/solanaos-mobile-<version>-debug.apk
```

## Mobile Wallet Adapter

The app includes native Solana Mobile Wallet Adapter support.

Current wallet actions include:

- connect wallet
- disconnect / revoke auth
- Sign In With Solana
- detached message signing
- sign unsigned transaction payloads
- sign and send transaction payloads

The Connect surface and raw transaction tools are intended for real Seeker-device validation.

For device testing:

- install an MWA-compatible wallet such as Solflare or Jupiter Mobile
- or use the Solana Mobile mock wallet during development
- test connect, SIWS, detached message signing, and raw transaction signing/sending from the app

## Shared Setup Code

Local runtime installs generate a shared handoff bundle at:

```bash
~/.nanosolana/connect/
```

Useful files:

```bash
cat ~/.nanosolana/connect/setup-code.txt
cat ~/.nanosolana/connect/solanaos-connect.json
```

Use that setup code in:

- onboarding
- the Connect tab manual setup flow

If you want Android to import a LAN/manual host instead of the Tailscale-advertised address, regenerate the bundle after starting the runtime with:

```bash
solanaos gateway start --no-tailscale
```

If you change the port, regenerate the setup bundle after:

```bash
solanaos gateway start --port <port>
```

## Convex Backend

The Android app now uses Convex for more than wallet sync.

Current production defaults in the Android build:

- cloud URL: `https://artful-frog-940.convex.cloud`
- site URL: `https://artful-frog-940.convex.site`
- health endpoint: `https://artful-frog-940.convex.site/nanosolana/health`
- wallet sync endpoint: `https://artful-frog-940.convex.site/nanosolana/users/upsert`

Current Convex-backed uses include:

- wallet-backed user registration and sync
- pairing claim/session state
- wallet-agent registry reads from `/nanosolana/agents/mine`
- public NanoHub agent profile / A2A / explorer link resolution through shared agent records
- Grok gallery feed and rating
- shared backend identity for Seeker-linked users
- chess archive/state
- token-analysis and tracker-adjacent backend flows

Important:

- only public Convex URLs ship in the APK
- deploy keys stay in CI/deploy tooling
- wallet identity sync is signature-based

To override the default deployment for a build:

```bash
export SOLANAOS_ANDROID_CONVEX_URL="https://<deployment>.convex.cloud"
export SOLANAOS_ANDROID_CONVEX_SITE_URL="https://<deployment>.convex.site"
cd apps/android
GRADLE_USER_HOME="$PWD/.gradle-user" ./gradlew :app:assembleDebug
```

## Agent Registry

The Android app now surfaces the shared Hub registry state inside `Connect`.

Current behavior:

- after wallet sync, Seeker reads the user's agent records from Convex
- agent rows reflect `8004`, `metaplex`, or `dual` registry mode
- each row exposes public NanoHub profile, A2A card, explorer links, and ACP command metadata
- Hub-created agents advertise the canonical ACP command `solanaos acp`

Public Hub surfaces are rooted at:

- `https://seeker.solanaos.net`
- `https://seeker.solanaos.net/agents/<AGENT_ID>`
- `https://<convex-site>/nanosolana/agents/registration?id=<AGENT_ID>`
- `https://<convex-site>/nanosolana/agents/agent-card?id=<AGENT_ID>`

No. This app cannot run in Expo Go.

Reasons:

- it is a native Kotlin Android project

- it uses Solana Mobile Wallet Adapter directly

## SolanaOS CLI On Device

Once installed, the SolanaOS binary on-device can run the same core commands you use elsewhere:

| Command | Description |
| --- | --- |
| `solanaos seeker` | Start the Android Seeker runtime |
| `solanaos solana health` | Check RPC health |
| `solanaos solana balance` | Check wallet balance |
| `solanaos ooda --sim` | Run paper OODA mode |
| `solanaos daemon` | Run the full daemon |
| `solanaos pet` | Show companion status |
