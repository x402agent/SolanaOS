SolanaOS — The Solana Computer Guide

A local-first operating layer for Solana trading, research, wallets, automation, and operator control

SolanaOS is a modular, security-first runtime for building autonomous financial systems on Solana. Run one operator-grade Go daemon for trading, memory, skills, Telegram control, remote gateway access, and companion hardware.

1. Why SolanaOS?

The financial world is being rebuilt by autonomous agents. But today's agent frameworks are fundamentally flawed for finance:

  • Built for chat, not finance — retrofitting chatbots for trading is dangerous
  • Stateless — they forget every trade, every lesson, every pattern
  • Siloed — each agent is an island with no coordination
  • Insecure — API keys in .env files, no encryption, no audit trail

SolanaOS is built from the ground up for financial agents:

  • OODA Trading Loop — military-grade decision cycle (Observe → Orient → Decide → Act → Learn)
  • Epistemological Memory — 3-tier ClawVault that distinguishes facts from patterns from hypotheses
  • Mesh Coordination — agents share signals and lessons across a Tailscale VPN mesh
  • Vault-Encrypted Secrets — AES-256-GCM for every API key and private key, always
  • On-Chain Identity — every agent mints a Metaplex NFT birth certificate at creation

SolanaOS ships in two implementations: a full-featured TypeScript runtime (nano-core) for rapid development and extensibility, and an ultra-lightweight Go runtime that runs on anything from an NVIDIA Orin Nano to a Raspberry Pi in under 10MB.

3. Architecture & Core Concepts

System Overview

SolanaOS is a modular runtime for deploying autonomous financial agents on Solana. The system is organized into three layers:

layers
┌─────────────────────────────────────────────────────────┐
│                    AGENT RUNTIME                        │
│         OODA Loop · ClawVault · Strategy Engine         │
├─────────────────────────────────────────────────────────┤
│                  INFRASTRUCTURE                         │
│     Vault · Gateway · Mesh Network · On-Chain ID        │
├─────────────────────────────────────────────────────────┤
│                    INTERFACES                           │
│   CLI · Telegram · Discord · SolanaOS Control · Chrome Ext    │
└─────────────────────────────────────────────────────────┘

Agent Runtime — the brain. Houses the OODA trading loop, epistemological memory, AI reasoning, and strategy execution.

Infrastructure — the skeleton. Encrypted secret storage, authenticated gateways, peer-to-peer mesh networking, and Solana wallet/NFT identity.

Interfaces — the skin. Every surface a human (or another agent) can use to interact with a running SolanaOS instance.

4. The OODA Trading Loop

OODA loop
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ OBSERVE  │──▶│  ORIENT  │──▶│  DECIDE  │──▶│   ACT    │
│ Helius   │   │ OpenRouter│   │ Signal   │   │ Jupiter  │
│ Birdeye  │   │ AI Model │   │ Scoring  │   │ Swaps    │
└──────────┘   └──────────┘   └──────────┘   └────┬─────┘
     ▲                                             │
     │              ┌──────────┐                   │
     └──────────────│  LEARN   │◀──────────────────┘
                    │ClawVault │
                    └──────────┘

Phase 1 — Observe

Pulls real-time data: Helius RPC on-chain state, Birdeye token prices/volume/liquidity, WebSocket streaming prices, and wallet state. All enters ClawVault at the KNOWN tier with a 60-second TTL.

Phase 2 — Orient

Raw observations fed to AI model via OpenRouter alongside ClawVault memory, strategy parameters, companion runtime state, recent trade history, and the agent's SOUL.md system prompt.

Phase 3 — Decide

Strategy engine combines AI orientation with quantitative signals. Confidence scoring: RSI strength (30%), EMA crossover (30%), Volume confirmation (20%), Memory pattern match (20%). Only signals with confidence ≥ 0.7 advance to execution.

Phase 4 — Act

Executed via Jupiter Ultra Swap with dynamic slippage (ATR-based), Kelly Criterion sizing, hard limits (50% max position, -10% daily loss breaker), companion runtime state modifier, and 0.01 SOL minimum gas reserve.

Phase 5 — Learn

After every trade: reviews last 20 outcomes, promotes recurring patterns to LEARNED tier, generates tentative correlations in INFERRED tier, runs contradiction detection, updates research agenda, and broadcasts lessons to mesh peers.

OperationTimeout
Full OODA cycle600s
External API call30s
AI inference120s
Trade execution60s

5. ClawVault: Epistemological Memory

Most agent frameworks use flat context windows. ClawVault introduces epistemological tiers that model how confident the agent is in each piece of knowledge.

Memory Tiers

TierTTLStatusExample
KNOWN60 secondsEmpirical fact — directly observed"SOL is at $142.50 right now"
LEARNED7 daysValidated pattern — confirmed across trades"RSI < 30 + volume spike → 72% bounce rate"
INFERRED3 daysHypothesis — tentative correlation"This token might correlate with BTC moves"

Key Mechanisms

  • Temporal Decay — every memory has a TTL. Garbage-collected every 5 minutes.
  • Experience Replay — reviews last 20 outcomes after each trade. Promotes patterns from INFERRED to LEARNED.
  • Contradiction Detection — new KNOWN data contradicts INFERRED hypothesis? Dropped immediately.
  • Research Agenda — open questions the agent prioritizes during data collection.

Memory Flow

memory flow
  Observe (API data)
       │
       ▼
   ┌────────┐  60s TTL
   │ KNOWN  │────────────────▶ expired → garbage collected
   └───┬────┘
       │ experience replay
       ▼
   ┌────────┐  3d TTL
   │INFERRED│────────┬───────▶ contradicted → dropped
   └───┬────┘        │
       │ validated   │ expired → garbage collected
       ▼             │
   ┌────────┐  7d TTL
   │LEARNED │────────────────▶ expired → garbage collected
   └────────┘

5A. Honcho Solana Memory

ClawVault handles local epistemological memory. Honcho adds a second layer: durable, session-oriented social memory for the Telegram trading bot. It stores conversations as workspace sessions, tracks peers, generates summaries, and promotes repeated user behavior into reusable conclusions.

What Is Stored

ObjectHow SolanaOS Uses It
WorkspaceSingle memory domain for the Solana trading bot
SessionPer-chat or per-trading-thread message timeline with trading-aware configuration
PeersUser peer and agent peer with metadata, peer cards, and observation config
MessagesUser turns, bot replies, optional files, and per-message reasoning configuration
SummariesShort and long rollups used for prompt recall and session compression
ConclusionsExplicit memory facts like preferred tokens, spot bias, perps usage, and risk tolerance

Trading Adaptation

  • Session metadata includes wallet, watchlist, mode, and strategy context.
  • Peer metadata models the human trader separately from the SolanaOS agent.
  • Conclusion heuristics infer token interests, spot vs perps preference, and research style.
  • Prompt enrichment injects summaries and semantic conclusion matches before replying.

Memory Flow

honcho flow
Telegram turn
    │
    ├── store message in session
    ├── update user + agent peer state
    ├── refresh short / long summaries
    ├── generate trading conclusions
    └── recall matched conclusions into next prompt

Examples:
- "I mostly scalp BONK and WIF"        → preferred tokens
- "I only want spot, no perps today"   → spot-only bias
- "Size me small, I am risk-off"       → conservative risk posture

Bot Commands

honcho commands
/honcho_status            bridge + workspace status
/honcho_context           current session context with summaries
/honcho_sessions          list sessions in the workspace
/honcho_summaries         short + long summaries for the session
/honcho_search <query>    search messages inside the session
/honcho_messages          list recent stored messages
/honcho_message <id>      inspect a single stored message
/honcho_conclusions       query recalled trading conclusions

6. Trading Engine & Strategy

Signal Generation

LONG when:

  • RSI crosses above oversold (< 30)
  • Fresh bullish EMA crossover
  • Price above fast EMA
  • ATR confirms sufficient volatility

SHORT when:

  • RSI crosses below overbought (> 70)
  • Fresh bearish EMA crossover
  • Price below fast EMA
  • OR stop-loss / take-profit hit

Risk Management

ControlValue
Max position50% of wallet balance
Daily loss limit-10% → trading paused 24h
Max slippage3% hard cap
Minimum reserve0.01 SOL for gas
Position sizingKelly Criterion adaptation
Companion runtime stateModifies position size ±30%

⚠️ Trading is disabled by default (execution.enabled: false). Auto-execution requires explicit opt-in.

7. Companion Runtime

Every SolanaOS runtime includes a companion state layer tied to the agent wallet. It is not just cosmetic; it is a risk management primitive.

evolution
🥚 Egg  →  🐛 Larva  →  🐣 Juvenile  →  🦞 Adult  →  👑 Alpha
                                                          │
                          👻 Ghost  ◀── (health = 0) ─────┘

Mood × Risk

MoodTriggerRisk Effect
😊 HappyRecent wins+10% position size
😐 ContentNormalNo change
🤤 HungryNot fed in 24h-10% position size
😢 SadRecent losses-15% position size
🤒 SickLosses + hunger-30% position size
👻 GhostHealth = 0Trading disabled

This creates a forcing function: neglect your agent, and trading progressively de-risks and eventually halts. A dead man's switch built into the design.

9. On-Chain Identity

Every agent mints a Metaplex NFT birth certificate on Solana devnet at creation — a tamper-proof record of when and how the agent was created. solanaos solana register mints a separate identity NFT with the agent's public key, version, registered skills, and SHA-256 fingerprint.

10. Mesh Networking

Agents form a peer-to-peer mesh network over Tailscale VPN. Shared: trading signals, learned patterns, price feeds, pet status. NEVER shared: wallet keys, private keys. Setup: install Tailscale, configure peers in config.json, run nanosolana gateway run --bind tailnet.

11. Gateway Architecture

A single long-lived SolanaOS Gateway owns all communication surfaces. WebSocket + text frames with JSON payloads. First frame must be auth with HMAC-SHA256 signature. HTTP endpoints at /api/status, /api/framework, /api/memory require X-SolanaOS-Secret header.

12. Sessions & Persistence

Session types: Main, Telegram DM/group, Discord, Nostr, Trading. Persisted to ~/.nanosolana/sessions/ as JSONL. Telegram stores up to 200 messages per chat with auto-summarization. Idle sessions expire after 7 days.

13. Multi-Channel Communication

14+ channel plugins: Telegram (full persistence), Discord, Nostr, iMessage, Google Chat, Web UI, Slack, Matrix, Signal, WhatsApp, Teams, and more. Build your own with the plugin SDK. Telegram is built-in with zero-dep HTTP client, long polling, and allowlist filtering.

14. SolanaOS Control Interactive UI

Floating interactive widget at localhost:7777. Tabs: Home (quick actions), Wallet (live SOL balance, DAS portfolio, send SOL, tx history), Chat (talk to SolanaOS Control), Tools (on-chain registration, system status). API endpoints: /api/wallet, /api/wallet/portfolio, /api/wallet/tokens, /api/wallet/history, /api/wallet/send, /api/chat, /api/run, /api/status.

15. Chrome Extension

Manifest V3 extension: tab relay via CDP, wallet panel, chat relay (forwarding to Telegram), manual trade submission with confidence scores, and gateway config sync. Install: load unpacked from chrome-extension/. Features: live SOL balance with USD, chat from toolbar, trending tokens, green/red badge for server status.

16. Go Runtime Implementation

Ultra-lightweight pure Go binary: 10MB, <10MB RAM, 1s boot. Cross-compiles to x86_64, ARM64 (Orin/RPi), RISC-V. Native Solana SDK (gagliardetto/solana-go). 20+ core packages: daemon, agent, nanobot, solana, onchain, tamagochi, strategy, hardware, channels, gateway, x402, memory, bus, config, aster, hyperliquid.

17. Hardware Integration (Arduino Modulino®)

7 sensors over a single Qwiic/I2C cable: Pixels (8× RGB LED, 0x6C), Buzzer (0x3C), Buttons (3×, 0x7C), Knob (rotary encoder, 0x76), Thermo (0x44), Distance (ToF, 0x29), Movement (6-axis IMU, 0x6A). Supported on NVIDIA Orin Nano, Spark, Raspberry Pi, any Linux SBC. macOS/Windows runs in stub mode. All hardware gracefully degrades — no sensors? Zero errors.

18. x402 Payment Protocol

Crypto-gated HTTP APIs via x402.org. Solana USDC payments via agent wallet (auto-configured SVM signer). Multi-chain: Solana, Base, Polygon, Avalanche. HTTP middleware for paywalling endpoints. Connects to facilitator.x402.rs. X402_PAYWALL_ENABLED=true ./build/solanaos daemon

19. Platform Apps

macOS: dist/SolanaOS-v2.0.0.dmg — native menu bar agent with quick status control, wallet access, trending tokens, OODA loop toggle. Build with bash scripts/package-macos.sh.

Android — SeekerClaw: Android 14+ foreground service agent for Solana Seeker devices. Kotlin, Telegram remote control, native Solana wallet, Claude-powered AI, secure on-device key management.

21. Deployment & Infrastructure

Docker: make docker → ~15MB image. Cross-compile: make orin (ARM64), make rpi, make riscv, make macos, make cross (all). Brev.dev: brev create nanosolana --gpu. systemd: unit file included for Linux/Orin Nano. Remote access: loopback-first SSH gateway plus Tailscale Serve/Funnel for secure control UI exposure. Database: optional Supabase (PostgreSQL + pgvector) at schema.sql.
